# 架构决策记录（ADR）

> 2026-07-16。基于 [需求文档v2-功能设计优化.md](需求文档v2-功能设计优化.md)。已确认：全新代码库、不参考旧系统实现、对话 LLM 走厂商配置（支持多模型切换）。

## ADR-1 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 仓库 | pnpm workspaces 单仓（monorepo） | 前后端共享 schema/类型（能力描述、实体校验），一次提交跨端一致 |
| 语言 | TypeScript 全栈 | 前后端同语言，共享 zod schema 即运行时校验 + 静态类型 |
| 后端 | Node.js + Fastify | 轻量、快、生态成熟；I/O 密集（编排外部 API/GPU）场景 Node 合适 |
| ORM/DB | Prisma + SQLite（开发）→ PostgreSQL（生产） | 开发零依赖（Windows 免装服务）；Prisma 换 datasource 平滑迁移。SQLite 下枚举/JSON 用 String 列 + 共享 zod 校验兜底 |
| 任务队列 | 自研 DB 队列（Job 表 + 进程内 worker 轮询） | 任务本质是长轮询外部 API/GPU，DB 队列足够且免 Redis 依赖；Job 表同时就是监控/统计的数据源（v2 §7） |
| 前端 | Vite + React 18 + Ant Design 5 + TanStack Query + Zustand + react-router | 管理密集型 UI，AntD 组件覆盖度高、中文生态好 |
| 媒体处理 | FFmpeg（服务端） | 拼接/合成/抽帧/抽音/波形/缩略图全部覆盖；开发机已装 |
| GPU 集群 | Python Agent + ComfyUI（心跳注册，沿用旧系统模式） | 旧模式已验证：Agent 只需能访问平台域名，任务数据下发、结果回传（里程碑 3） |
| LLM 接入 | OpenAI 兼容协议适配器 | DeepSeek/通义/豆包/Kimi 等国内厂商均兼容该协议，一个适配器覆盖多厂商，多模型切换=切 ModelConfig |

## ADR-2 剪辑器选型（问题 4 的回答）

**结论：渲染引擎自持（服务端 FFmpeg），时间轴 UI 自研；v2.1 阶段再评估是否引入 OpenCut。**

- **合成/导出永远走后端 FFmpeg Job**（filter_complex 拼接、音轨混合、导出参数），不依赖浏览器编解码——这保证任何 UI 方案下成片能力都在。
- **v2.0 自研轻量时间轴**（React 组件）：调序、删段、段间硬切、音轨对齐、单段增强入口。配音页与分镜页本来就需要时间轴组件，三处共用一套轨道模型。
- **v2.1 候选**：
  - **OpenCut**（MIT 许可，Web 剪辑器，浏览器内预览）——首选集成评估对象；
  - Remotion —— 排除：定位是程序化视频框架且公司使用需商业许可；
  - 各类商用 react-video-editor 组件 —— 许可不干净，排除。
- 判断依据：平台对剪辑的真实诉求是"拼接 + 对齐 + 增强 + 导出"（v2 §10），不是剪映全功能。轨道模型与渲染指令分离设计下，自研时间轴扩展到 v2.1 的成本可控，届时再对比 OpenCut 集成成本做决定。

## ADR-3 单仓布局

```
Ovideo/
├─ pnpm-workspace.yaml
├─ packages/shared/        # zod 实体 schema、能力描述、API 类型、分镜 patch 协议
├─ apps/server/            # Fastify + Prisma；modules/ 按业务域分（project/asset/job/provider/stale/...）
├─ apps/web/               # Vite + React + AntD；pages/ 按 九阶段+管理后台 分
├─ agents/comfyui-agent/   # Python GPU Agent（里程碑 3）
└─ docs/                   # 需求、架构、计划
```

## ADR-4 里程碑划分

| 里程碑 | 内容 | 对应 v2 |
|---|---|---|
| **M1 平台地基**（当前） | 单仓脚手架、全量数据模型、资产库+血缘、Job 系统+Mock 执行器（FFmpeg 合成占位媒体）、厂商配置+能力描述+OpenAI 兼容文本适配器、失效传播引擎、剧本三步生成（patch 语义）、九阶段前端骨架、项目/分集/剧本/管理后台页面 | §1 §2 §4(patch) §7 §8 |
| M2 生成管线 | 真实图像/视频/TTS 厂商适配器、设计/素材/配音/分镜/视频五阶段完整交互、抽卡画廊、时长链路、绑定联动 | §3 §9 |
| M3 衔接与集群 | ShotGroup 首尾帧衔接、GPU Agent + ComfyUI 工作流执行器、美化 v2.0（拼接+增强+导出）、成品页 | §5 §10 |
| M4 体验完善 | 对话式剧本交互（LLM patch 流）、声音一致性双轨、共享反馈、素材库预览、历史筛选 | §4 §6 |

M1 验收：`pnpm dev` 一条命令起前后端；创建项目→分集→贴剧本→三步生成出分镜卡片（Mock 或真实 DeepSeek）→Job 面板可见任务流转；核心规则（失效传播、绑定解析、patch 应用）有单测覆盖。
