# AI 漫剧创作平台 v2 —— 功能设计优化

> 本文档是对 [需求文档-AI漫剧创作平台重构.md](需求文档-AI漫剧创作平台重构.md)（v1，视频走查记录）的功能性深化。
> v1 回答"要做什么"，本文档回答"按什么规则做"——把 v1 中含糊、缺失、互相冲突的功能点落成可开发的明确规则。
> 安全/积分体系不在本文档展开（按 v1 §4.2 保留即可）。

---

## 0. 六条核心设计原则

后面所有规则都由这六条推导而来，开发中遇到没写到的情况，按原则裁决：

1. **一切产物皆资产（Asset），一切资产有血缘（Lineage）**。图片、视频片段、音频、帧、声音样本、成片，统一入资产库；每个资产记录"由什么资产 + 什么参数 + 哪个任务"生成。全平台只存一份，各处引用。
2. **标签（Tag）属于项目，绑定（Binding）属于镜头**。标签是一致性的锚点，跨分集共享；具体某镜头用哪张图是镜头级决策。
3. **付费产物从不自动删除、从不自动重生成**。上游变更只把下游**标记为 stale** 并提示，重生成永远由用户确认（可单个、可批量）。
4. **多次生成是常态（抽卡），"选中"是唯一状态**。每个产物槽可以有 N 个 take，有且只有一个 selected；切换 selected 即换片，历史 take 永远可回切。
5. **时长自下而上确定**：手动设定 > 配音实际时长 > 文本估算。时长一旦锁定，向下游（分镜、视频、剪辑）单向流动。
6. **对话用于"生成与修改内容"，结构化 UI 用于"绑定与确认"**。不是所有页面都改成聊天。

## 1. 核心数据模型

这是 v1 最大的缺口。以下实体定义是全系统的骨架（字段列关键项，非穷举）：

```
Project 项目
 ├─ members[]                    # 成员即权限（v1 §3.1 的可见性改造）
 ├─ Tag 标签 (project 级)         # type: character/scene/prop; name 项目内唯一
 │   └─ designs[] → Asset        # 该标签的候选设计图，其一为 canonical（默认参考）
 ├─ Asset 资产 (project 级)       # type: image/video/audio/frame/voice_sample/final
 │   ├─ source: generated / uploaded / extracted(抽帧·抽声)
 │   ├─ lineage: parent_asset_ids[], job_id, model, prompt, seed, params
 │   └─ status: active / recycled
 └─ Episode 分集
     ├─ ScriptDraft 剧本稿[]      # 多版本；恰好一个 is_main（主剧本）
     ├─ Storyboard 分镜脚本[]     # 版本化；由某个 ScriptDraft 生成
     │   └─ Shot 镜头[]
     │       ├─ sort_order, source_text(原文片段)
     │       ├─ image_prompt, video_prompt, dialogue_lines[]
     │       ├─ tags[] → Tag                    # 引用项目级标签
     │       ├─ duration_planned / duration_locked
     │       ├─ group_id → ShotGroup?           # 长镜头衔接组（见 §5）
     │       ├─ keyframe_slot:  takes[] → Asset, selected_id, stale?
     │       └─ video_slot:     takes[] → Asset, selected_id, stale?
     ├─ Binding 绑定[]            # (tag_id, episode) → asset_id 为标签级默认
     │                            # + (shot_id, tag_id) → asset_id 镜头级覆盖
     ├─ DubbingLine 配音行[]      # shot_id, speaker(tag|narrator), text, voice_id,
     │                            # speed, audio → Asset, duration
     └─ Cut 成片[]                # 有序 selected video takes + 音轨 + 版本历史
Job 任务                          # 见 §7
VoiceProfile 角色声音             # project 级；tag_id ↔ TTS 音色 / 声音样本资产
```

**由模型直接解决的 v1 问题：**

| v1 的问题 | 模型层面的解 |
|---|---|
| 新分集要"同步已有设计"按钮 | Tag/Asset 本来就是项目级，分集只是引用——不需要同步，天然可见 |
| 素材库"本集/全部"切换乱 | 资产带引用关系，按"被本集引用"过滤即是本集素材 |
| 参考图上传后要双向同步回设计/素材 | 上传即入项目资产库并挂到对应 Tag 下，各处引用同一条记录 |
| 换绑沙发后生成仍用旧沙发（Bug 6） | 生成任务的输入必须从 Binding 实时解析，禁止任务创建时快照绑定 |
| 历史/抽卡/成品用料追溯 三处各自实现 | 全部是 lineage 的查询视图 |
| 多剧本稿切换丢上下文 | 产物挂 Storyboard/Shot，Storyboard 挂 ScriptDraft；切主剧本不动资产 |

## 2. 失效传播规则（Staleness）——本次补充的最关键规格

v1 完全没有定义"上游改了，下游怎么办"。规则如下：

### 2.1 依赖链

```
ScriptDraft → Storyboard → Shot(prompts/tags) ┬→ keyframe_slot → video_slot → Cut
                                              │        ↑              ↑
                          Binding(标签→资产) ──┘        │              │
                          DubbingLine(时长) ───────────┴──────────────┘
```

### 2.2 传播表

| 变更事件 | 影响范围 | 动作 |
|---|---|---|
| 修改剧本稿内容 / 切换主剧本 | 该稿关联的 Storyboard | Storyboard 标 stale，提示"重新生成或对话修改"；**资产、标签、绑定一律不动** |
| 对话修改分镜（镜头增/删/改） | 仅被修改的 Shot | 改 prompt → 该镜头 keyframe+video 标 stale；删镜头 → 产物进回收站；新镜头 → 空槽。**未被触及的镜头零影响**（修复 v1 Bug 1 的"重复累加"：对话输出为 patch，见 §4） |
| 标签级绑定变更（换角色图/场景图） | 本集所有含该标签、且无镜头级覆盖的 Shot | 这些镜头的 keyframe 标 stale；联动提示"N 个镜头受影响，是否批量重生成（预览列表）" |
| 镜头级绑定覆盖 | 仅该 Shot | 该镜头 keyframe 标 stale |
| 配音重生成导致 duration_locked 变化 | 该 Shot 的 video_slot；Cut | 时长差 > 阈值(0.5s) → video 标 stale；Cut 时间轴自动重排（拼接不花钱，可自动） |
| keyframe 换 selected take / 重生成 | 该 Shot 的 video_slot | video 标 stale |
| video 换 selected take | Cut | Cut 自动重排（自动，不花钱） |
| 设计图（Tag canonical）更换 | 各集的标签级 Binding | 绑定**不自动跟随**，在绑定处显示"该标签有新设计"提示，用户确认后才换绑 |

### 2.3 stale 的呈现与消费

- 镜头卡片右上角黄色角标"上游已变更"，点开显示**变更溯源**（谁、什么时候、改了什么导致的）。
- 两种消费方式：**重新生成**（消 stale）或 **忽略**（显式接受现状，角标消失但溯源记录保留）。
- 全局"待重生成"面板：列出本集所有 stale 项，勾选批量重生成。
- stale 只是标记，**不阻塞**任何下游操作（用户可以带着 stale 继续拼片）。

## 3. 时长链路规则

v1 说"全链路联动"但没给规则。定义如下：

1. 每个 Shot 的时长有两个字段：
   - `duration_planned`：分镜生成时的文本估算值（默认 10–15s，见拆分策略）。
   - `duration_locked`：配音阶段确定的实际值 = `max(该镜头全部配音行时长之和 + 间隔, 手动设定最小值)`。手动设定永远最高优先。
2. 配音生成/语速调整/重新生成 → 重算 `duration_locked` → 按 §2.2 传播（修复 v1 Bug 4：时长不扩展）。
3. `duration_locked` 超过所选视频模型的单次上限（模型能力里声明，如 Seedance 15s / 海螺 10s）→ 系统提示并**自动建议拆分为衔接组**（§5），用户一键接受或手动调整。
4. 配音页时间轴 = 唯一时长事实源的编辑器：每行显示镜头缩略图 + 对白/旁白文本 + 音频波形，行时长即 `duration_locked`，可拖拽微调（写回手动值）。上方逐句列表与时间轴是同一数据的两个视图（修复 Bug 5 的对不齐——不允许两份数据）。
5. 美化/剪辑时间轴按 selected video take 的**实际时长**排布；与 `duration_locked` 不符时该片段显示时长偏差提示。

## 4. 剧本阶段：对话式交互的工程边界

对话模式保留（v1 的核心诉求），但必须加三条硬约束，否则标签体系和镜头产物会被 LLM 冲垮：

1. **结构化 patch 输出**：对话的每轮产出不是自由文本，而是 schema 约束的 Storyboard JSON **补丁**（add_shot / remove_shot / update_shot / reorder）。应用补丁 → 生成新 Storyboard 版本。这从机制上消灭"重新生成在镜头 11 后累加"（Bug 1）：LLM 永远输出针对当前版本的变更集，而非整份重写。
2. **标签词表注入**：每轮对话上下文携带项目现有标签清单，指令要求复用；LLM 新造的标签标记为 `pending`，在结果面板里高亮，用户确认"新建标签"或"合并到现有标签"后才入库。
3. **diff 预览后应用**：每轮生成先展示与当前版本的镜头级 diff（新增/删除/修改高亮），用户点"应用"才落版本。防止一句"帮我精简一下"悄悄改掉已生成产物的镜头。

页面结构维持 v1 描述：左侧剧本稿列表（主剧本置顶标星）→ 中间对话窗 → 右侧可折叠资产面板（分镜卡片，标准卡片式、就地编辑、支持中间插入）。视频反推作为对话窗的一个工具按钮（上传视频 → 产出剧本稿草稿），不再是独立页面。

**拆分策略（写进生成 prompt 的硬规则）**：同场景无转场的连续剧情合并为 10–15 秒/镜头；有转场处必须切镜头；每镜头必须产出四件套（标签 / 对白 / 生图 prompt / 视频 prompt）。

## 5. 长镜头衔接组（ShotGroup）——从"操作技巧"升级为一等功能

v1 里首尾帧衔接是用户手工操作（抽尾帧→存素材→下段当首帧）。v2 建模为 **ShotGroup**：

1. 触发：`duration_locked` > 模型单次上限，或用户手动把连续镜头标记为组。
2. 系统自动把该镜头拆为 segment 1..n（每段 10–15s），共享同一场景绑定与人物绑定。
3. **生成强制串行**：segment i 生成完成并选定 take 后，自动抽取尾帧（入资产库，lineage 指向该 take），作为 segment i+1 的首帧输入；i+1 的任务卡片显示"等待上一段选定"。
4. 换 segment i 的 selected take → i+1 之后全部标 stale（尾帧变了）。
5. **漂移控制**：组视图显示各段首帧缩略图排成一行，肉眼可查色偏/画质衰减；段数 > 4 时提示质量风险；每段生成参数里默认附带该镜头 keyframe 作为风格参考（模型支持多参考图时），抑制漂移。
6. 手动裁帧能力保留：任一段可裁掉闪动帧/多余秒数，裁剪产生新 take（lineage 指向原 take），尾帧从裁后版本抽取。

（修复 v1 Bug 7 即以此功能形态交付，不修旧交互。）

## 6. 声音一致性：主方案 + 降级方案

v1 的"抽声音样本传给视频模型"依赖厂商能力，不可承诺为通用功能。定义为双轨：

- **主方案（默认，全厂商可用）**：对白统一由 TTS 产出（VoiceProfile：每个角色标签绑定一个音色，项目级）。视频生成不依赖模型自带人声；美化阶段用**对口型**能力以 TTS 音频驱动口型。全片声音一致性由 TTS 保证。
- **增强方案（能力开关）**：厂商能力描述（§8）声明 `supports_voice_reference` 的模型，视频生成参数面板出现"声音样本"输入；声音样本 = 从既有片段**抽取的音频资产**（与抽帧同级的工具：选中片段 → 框选时间段 → 抽取为 voice_sample 资产，挂到角色标签）。
- 声音样本资产同时可用于支持克隆的 TTS 厂商（VoiceProfile 的另一种来源）。

## 7. 任务系统（Job）——升级为 P0 一等公民

全平台的生成动作（生图/生视频/TTS/反推/放大/补帧/对口型/抠图/角度）统一走一个 Job 模型，这是进度反馈、抽卡、重试、监控的共同底座：

- **状态机**：`queued → dispatched → running(progress%) → succeeded / failed / canceled`。
- **执行器两类**：API 厂商（HTTP 轮询或回调）、GPU 集群节点（ComfyUI，WebSocket 进度直推）。对上层同构。
- **输入解析时机**：任务**出队执行时**才解析 Binding/参数（修复 Bug 6 的旧值快照问题）。
- **产出**：成功 → 产 Asset（带 lineage）挂到目标槽位为新 take；失败 → 槽位显示失败卡片 + 错误信息 + 一键重试（保留 v1 的"失败自动重试 1 次"为可配置项）。
- **进度体验**：所有生成位点显示占位卡（排队中/执行中 N%/预览图若厂商支持）；页面级不弹窗，完成一律 toast + 槽位刷新（v1 明确诉求）。
- **批量任务组**：如"重生成本集全部 stale 关键图"= 一个 batch，面板显示成员进度、可整组取消。
- 每个 Job 记录 provider、model、耗时、结果——v1 后台的任务监控/生成统计/审计直接查询此表，无需另建。

## 8. 动态模型配置：用"能力描述"落地前后台联动

v1 只说"后台加模型前台出按钮"。机制上补齐为**能力描述（Capability Descriptor）**：

每个厂商配置下的每个模型声明：

```yaml
modality: text | image | video | tts
label: "海螺 I2V"            # 前台按钮名，后台可改
input:  [first_frame, first_last_frame, ref_images(max 4), voice_sample?]
output: { resolutions: [720p, 1080p], ratios: [16:9, 9:16, 1:1], max_duration_s: 10 }
params_schema: {...}          # JSON Schema，前台据此渲染参数表单
flags:  supports_voice_reference: false, supports_preview: true
```

- 前台每个"选模型"位点 = 按 `modality` + 所需 `input` 能力过滤出的模型列表，按钮与参数面板**全部由 descriptor 渲染**，前端零硬编码（"化神/筑基"档位彻底移除，直接显示 label + 模型名）。
- 模型不可用（连通失败/停用）→ 按钮置灰 + 原因提示（v1 §3.5 诉求）。
- **ComfyUI 工作流模板也是一种模型**：`modality` + 占位符即 `params_schema`，执行器为 GPU 集群。前台不感知"API 还是本地 GPU"的差异。
- 长镜头拆分（§3.3）、时长校验（§3）都从 descriptor 的 `max_duration_s` 读取，不写死。

## 9. 各阶段交互形态定稿（对话 vs 结构化）

| 阶段 | 形态 | 说明 |
|---|---|---|
| 剧本 | **对话窗**（§4） | 生成/修改文本内容，天然适合多轮 |
| 设计 | 结构化卡片 + 生成弹层 | 标签卡片墙；AI 生成弹层内可带一个"提示词助手"输入框（轻对话），但主体是表单选模型/比例/参考图 |
| 素材(绑定) | **纯结构化** | 绑定是精确操作，表单/拖拽最快；保留锁定联动、本集/全部切换 |
| 配音 | 结构化时间轴（§3.4） | 逐句列表与时间轴同数据双视图 |
| 分镜 | 结构化 + 图上工具 | 关键图槽位 + takes 画廊；抠图/转角度/姿势模仿等保留为图上工具 |
| 视频 | 任务卡片 + takes 画廊 | 每镜头一张任务卡（输入区/结果区），抽卡画廊选定 selected |
| 美化 | 时间轴剪辑器 | 见 §10 |
| 成品 | 展示画廊 + 版本列表 | 每版成片可展开"用料清单"（lineage 查询） |

通用组件：右侧**可折叠资产面板**（当前镜头/标签相关资产）贯穿所有阶段。

## 10. 美化/剪辑的分步交付

一步到位嵌完整剪辑器风险大，拆为两档：

- **v2.0（管线闭环所需的最小集）**：片段按 Cut 顺序自动排列的简单时间轴；调序、删段、段间硬切；音轨（配音轨自动对齐 + BGM 轨上传）；单段增强（放大/补帧/对口型，走 Job）；导出参数（分辨率/码率/格式）+ 合成。
- **v2.1（对标剪映的站内剪辑）**：多轨、裁切、转场、字幕、音量曲线、预览渲染。选型独立调研（候选：基于 WebCodecs 的开源编辑器内核，或服务端 FFmpeg 时间轴渲染 + 前端轻量编辑），**不阻塞 v2.0 上线**。

成品阶段的共享反馈闭环（分享链接 → 逐时间点评论 → 回流修改）保持 v1 描述，挂在 Cut 版本上。

## 11. 修订后的开发排期

| 阶段 | 内容 | 对应本文档 |
|---|---|---|
| **P0 · 骨架走通** | 数据模型 + 资产库/血缘 + Job 系统 + 能力描述式厂商配置 + GPU 集群执行器；用**现有三步生成交互**（修好 patch 语义）把 剧本→设计→素材→配音→分镜→视频→简版拼接导出 整条流水线走通一部片 | §1 §7 §8 + §4 的 patch 机制 |
| **P1 · 规则与体验** | 失效传播全量落地 + 时长链路 + 对话式剧本交互 + takes 抽卡画廊 + 配音时间轴重做 + ShotGroup 衔接组 | §2 §3 §4 §5 |
| **P2 · 增强** | 声音一致性双轨 + 剪辑器 v2.1 + 成品共享反馈 + 素材库预览/缩略图 + 历史筛选 + 工作流图形预览 | §6 §10 |
| **P3 · 外延** | 自由创作画布（标准 ComfyUI 前端）+ 前台回收站 + 费用余额看板 | — |

P0 的验收标准：**一个真实剧本，从贴入到导出成片，全程不离开平台、不手动搬运文件。**

## 12. 遗留的开放问题（较 v1 收敛后）

1. 旧系统源码/数据库可得性（决定数据模型是"迁移"还是"重建+导入"）。
